# Radius — Unified Session Ingestion Service

## Goal

Read sessions from Pi (JSONL files) and Opencode (SQLite), normalize into a single event-sourced SQLite DB, and expose queryable data for a dashboard.

---

## Architecture

```
features/
  sessions/
    schema.ts           ← Drizzle table definitions (type source of truth)
    service.ts          ← SessionDb Effect service (tag + interface + impl)
    layer.ts            ← Layer composition (DB connection + migration + service)
    ingest/
      pi.ts             ← Parse Pi JSONL → unified events
      opencode.ts       ← Read Opencode SQLite → unified events
      index.ts          ← Orchestrated ingest pipeline
    migrations/
      0000_*.sql        ← Generated by drizzle-kit
      meta/
        _journal.json
    index.ts            ← Barrel
```

Types are inferred from Drizzle via `typeof project.$inferSelect`. Pi and
Opencode raw data shapes are defined locally in ingest files — no shared
types.ts needed.

---

## Schema (Drizzle)

Three tables. Harness-agnostic — all payload lives in `data` JSON.

### `project`

One row per unique directory path.

| Column | Type | Notes |
|--------|------|-------|
| `id` | text PK | Normalized filesystem path |
| `name` | text | Display name from harness |
| `time_created` | integer NOT NULL | |
| `time_updated` | integer NOT NULL | |

### `session`

One row per conversation. Belongs to a project. `parent_session_id` handles forking/branching across both harnesses.

| Column | Type | Notes |
|--------|------|-------|
| `id` | text PK | Pi: uuidv7, Opencode: ses_xxx |
| `project_id` | text FK NOT NULL | |
| `parent_session_id` | text FK | Fork parent |
| `title` | text | Derived at ingest — latest `session_info.name`, or first ~80 chars of first user message |
| `directory` | text NOT NULL | CWD |
| `agent` | text | `"pi"` / `"opencode"` |
| `time_created` | integer NOT NULL | |
| `time_updated` | integer NOT NULL | |
| `time_archived` | integer | |

### `event`

Conversation events only. One row per message exchange. Content blocks (text,
tool calls, reasoning) are embedded in `data.content[]` for Pi or `data.parts[]`
for Opencode.

| Column | Type | Notes |
|--------|------|-------|
| `id` | text PK | Pi: short hex, Opencode: msg_xxx |
| `session_id` | text FK NOT NULL | |
| `parent_id` | text FK | Pi tree structure, NULL for root |
| `seq` | integer | Opencode only: step ordering within session |
| `event_type` | text NOT NULL | `message`, `step_start`, `step_finish` |
| `time_created` | integer NOT NULL | Unix ms |
| `data` | text NOT NULL | Full JSON payload |

### `session_event`

Session-level metadata mutations. Not conversation data — auxiliary signals
for deeper analysis. Same id/session_id/data structure but no seq or parent_id.

| Column | Type | Notes |
|--------|------|-------|
| `id` | text PK | |
| `session_id` | text FK NOT NULL | |
| `event_type` | text NOT NULL | See below |
| `time_created` | integer NOT NULL | Unix ms — ordering key |
| `data` | text NOT NULL | Full JSON payload |

Ordering is purely timestamp-based, unlike `event` which has tree structure
via `parent_id` and `seq`. Unix ms resolution is sufficient — no two metadata
mutations happen within the same millisecond in practice.

**event_type values**:

| event_type | Source | data contents |
|-----------|--------|---------------|
| `model_change` | Pi | `{"provider","modelId"}` |
| `thinking_change` | Pi | `{"thinkingLevel"}` |
| `compaction` | Pi | `{"summary","firstKeptEntryId","tokensBefore"}` |
| `branch_summary` | Pi | `{"fromId","summary"}` |
| `custom` | Pi | `{"customType","data"}` |
| `session_info` | Pi | `{"name"}` |
| `label` | Pi | `{"targetId","label"}` |
| `step_start` | Opencode | `{"snapshot"?}` — LLM round-trip start |
| `step_finish` | Opencode | `{"reason","cost","tokens":{...}, "snapshot"?}` — LLM round-trip end |

## Ingestion: Pi

**Input**: `~/.pi/agent/sessions/--<encoded-path>--/*.jsonl`

**Process**:

```
1. Scan session directories under ~/.pi/agent/sessions/
2. Effect.all(files, { concurrency: 10 }):
   a. FileSystem.readFileString → split by newline → parse JSON
   b. Line[0]: session header. Upsert project, upsert session
   c. Line[1..n]: each maps 1:1 to event
      - {"type":"message","message":{...}} → event_type=message, data=message
      - {"type":"model_change",...} → event_type=model_change
      - {"type":"thinking_level_change",...} → event_type=thinking_change
      - {"type":"compaction",...} → event_type=compaction
      - {"type":"branch_summary",...} → event_type=branch_summary
      - {"type":"custom",...} → event_type=custom
      - {"type":"session_info",...} → event_type=session_info
      - {"type":"label",...} → event_type=label
   d. parent_id = entry.parentId (preserves tree, NULL for root)
   e. Upsert event by id (INSERT OR IGNORE — Pi is append-only)
```

**Title derivation**: After ingesting all entries, scan for the latest
`session_info` entry. If it has a `name`, use that. Otherwise, find the first
`message` with `role="user"` and truncate its text to ~80 chars.

## Ingestion: Opencode

**Input**: `~/.local/share/opencode/opencode.db` (opened read-only via better-sqlite3)

**Process**:

```
1. Read all projects → upsert into unified project
2. Read all sessions → upsert into unified session
3. Read all messages (ordered by session_id, time_created):
   For each message:
     - Parse data JSON → role, model, provider
     - Build unified data: {role, content, model, provider, ...original extras}
     - Insert as event_type=message
   For each part that belongs to that message:
     - Merge into the message event's data.parts[]
     - Or: if the part represents a semantic event (step_start/finish),
       insert as separate event with parent_id = message.id
4. seq = sequential within each session (read from opencode's time_created order)
```

**Title derivation**: Opencode sessions have their own `title` — use it
directly. If empty, fall back to first ~80 chars of first user message.

**Flattening**: Parts of type `text`, `tool`, `reasoning` become entries in `data.content[]` or `data.parts[]` on the parent message event. Parts of type `step-start`, `step-finish` become separate events (they're control flow, not content).

## Effect Service (single file: `service.ts`)

```typescript
import { FileSystem } from "@effect/platform"

// Tag
export const SessionDb = Context.Service<SessionDb>("radius/SessionDb")

// Interface — every dependency is explicit in the Effect type signature
export interface SessionDb {
  // Lifecycle
  readonly migrate: Effect.Effect<void>
  readonly ingestPi: Effect.Effect<void, never, FileSystem>
  readonly ingestOpencode: Effect.Effect<void>
  readonly ingestAll: Effect.Effect<void, never, FileSystem>

  // Queries — no external deps, pure DB reads
  readonly getProjects: Effect.Effect<Project[]>
  readonly getSessions: (filter?: SessionFilter) => Effect.Effect<Session[]>
  readonly getEvents: (sessionId: string) => Effect.Effect<Event[]>
  readonly getModelUsage: Effect.Effect<{ model: string; count: number }[]>
  readonly getToolUsage: Effect.Effect<{ tool: string; count: number }[]>
}
```

Only `ingestPi` and `ingestAll` require `FileSystem` — the Opencode ingestion
reads a SQLite file directly via better-sqlite3, no async FS needed.

Implementation calls Drizzle directly — no wrapper:
```typescript
const make = (db: ReturnType<typeof drizzle>): SessionDb => ({
  getProjects: Effect.sync(() => db.select().from(project).all()),
  getSessions: (filter) =>
    Effect.sync(() => {
      const q = db.select().from(session).$dynamic()
      if (filter?.projectId) q.where(eq(session.projectId, filter.projectId))
      return q.all()
    }),
  // ...
})
```

`better-sqlite3` is synchronous, so `Effect.sync` suffices for reads. Writes
use `Effect.try` since `db.insert().run()` can throw on constraint violations
(handled by ON CONFLICT).

The service depends on `FileSystem` from `@effect/platform` (provided via
`NodeFileSystem.layer`) for async JSONL reads during ingestion. The layer
provides both `FileSystem` and the `SessionDb` service.

## Layer

```typescript
export const layer = (
  dbPath: string = join(homedir(), ".local/share/radius/sessions.db")
): Layer.Layer<SessionDb> =>
  Layer.scoped(
    SessionDb,
    Effect.gen(function*() {
      mkdirSync(dirname(dbPath), { recursive: true })
      const sqlite = new Database(dbPath)
      yield* Effect.addFinalizer(Effect.sync(() => sqlite.close()))
      sqlite.pragma("journal_mode = WAL")
      sqlite.pragma("foreign_keys = ON")
      const db = drizzle(sqlite, { schema })
      yield* migrate(db)
      return make(db)
    })
  )
```

Migration at startup: `drizzle-kit generate` produces SQL files; we run them via Drizzle's built-in migrator on start.

## Example Queries

```sql
-- Model usage (conversation events only)
SELECT json_extract(data, '$.model') as model, COUNT(*) as count
FROM event
WHERE event_type = 'message'
  AND json_extract(data, '$.role') = 'assistant'
GROUP BY model ORDER BY count DESC;

-- Tool calls (traverse content array on messages)
SELECT json_extract(part.value, '$.name') as tool, COUNT(*) as count
FROM event, json_each(json_extract(data, '$.content')) as part
WHERE event_type = 'message'
  AND json_extract(data, '$.role') = 'assistant'
  AND json_extract(part.value, '$.type') = 'toolCall'
GROUP BY tool ORDER BY count DESC;

-- Sessions per project per month
SELECT p.id as path, strftime('%Y-%m', s.time_created / 1000, 'unixepoch') as month,
       COUNT(*) as sessions
FROM session s JOIN project p ON s.project_id = p.id
GROUP BY p.id, month ORDER BY month DESC;

-- Model change frequency over time (session events)
SELECT json_extract(data, '$.modelId') as model, COUNT(*) as count
FROM session_event
WHERE event_type = 'model_change'
GROUP BY model ORDER BY count DESC;
```

## Resolved questions from design review

- **Purpose**: Personal analytics to understand AI coding session patterns
- **Project key**: filesystem path (no synthetic ID)
- **Event model**: conversation events (`event`) split from metadata events (`session_event`)
- **Pi seq**: not used — `parent_id` tree is the ordering
- **Opencode steps**: stored as separate `step_start`/`step_finish` events
- **Title derivation**: latest explicit name, else first ~80 chars of first user message
- **Packages**: no `@earendil-works/pi-*` — local types only
- **Filesystem**: `@effect/platform` FileSystem for async Pi JSONL reads
- **Dependencies**: explicit via Effect type signatures (`Effect<void, never, FileSystem>`)
- **Session version**: dropped (source format version irrelevant after ingest)
- **Project VCS**: dropped (not core to project identity)
- **Session slug**: dropped (display-only, not useful)
- **Session meta**: dropped (catch-all anti-pattern)
- **Ingest strategy**: full re-scan on each import (consistency over speed)
- **Upsert semantics**: Pi = INSERT OR IGNORE, Opencode = INSERT OR REPLACE
- **Cloudflare → Node.js**: local-only dashboard, Nitro node-server preset
